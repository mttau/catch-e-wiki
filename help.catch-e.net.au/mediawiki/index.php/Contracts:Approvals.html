Overview

Approvals are authorisations given for work to be carried out on a particular vehicle (or contract). Typically approvals are given out in a call centre environment. When a service provider (supplier) calls, the user searches the database for the vehicle (or contract) and creates a new approval if required. Existing approvals may be edited while they are in a pending status. Once a physical invoice is received from the supplier it is reconciled to the approval with information such as the invoice number added by the user. The approval can then be verified and posted. Once posted no further editing may be performed.

[edit]

Approval Status

Field Description
ActiveCreated but not yet posted
PostedReconciled to final invoice and posted for export to MYOB. Cannot be edited.
History OnlyCreated for reporting but not Accounting purposes (i.e. excluded from exports to accounting systems).
CancelledCreated but cancelled without saving.
DeletedDeleted sometime after creation but before posting.

Note that approvals are not physically deleted from the database and will appear in table data downloads but not in on line searches or reports.

Action Buttons

FieldStage Description
NewNewThe new button allows a new approval to be created. An approval number is displayed immediately which can be given verbally to the supplier. Details can be entered into the approval immediately or over several events by using the edit and save options.
EditExistingAn approval may be edited anytime it has an ‘active’ status. Changes should be saved by using the ‘Save’ button or alternatively changes can be ignored by selecting the ‘Cancel’ button when in edit mode.
SaveNew or EditThe approval needs to be saved when complete or edited
CancelNew or EditThe changes can be cancelled at any time
CheckSavedThe system will verify data entered before allowing the approval to be Posted.
AttachAny StageA document can be attached to an approval by clicking the Attach button. Attached approval documents are available to all user roles including external fleet managers and novated drivers.
ViewAny Stage View attached document by clicking on the paper clip icon . This is also available on the Maintenance screen.
PostCheckedIf check is completed successfully, the Post button becomes available. Once posted, the approval my no longer be edited and the relevant transactions become available for Recharge Invoicing and MYOB Export (supplier records).
HistoryCheckedWhen posting an approval it can be saved as ‘history’. This will not produce any accounting transactions nor a recharge invoice, nor will the transaction be included in any Account/export files. This option is to be used when an approval is not going to be paid or processed further, for example a commentary type transaction
Un-PostPostedAn approval may be un-posted and edited only if the approval has not had a recharge invoice created or the approval details have not been exported via the Accounts/Export/Purchase ledger function or by Data Interface. Unposting sets the Approval status to ‘active’ and deletes batch number references in the maintenance record.
DeleteUnpostedAn approval may be ‘deleted’ if it has not yet been posted. Note that the approval is stored by the system as ‘deleted’ and will not be used for accounting or reporting purposes.

Data Entry Validations

[edit]
Reminder Note

Reminder notes can be created for a Contract Type in the Contract Types / Details screen and a Client in the Clients / Client Defauts screen. These are displayed to the user when an Approval is created, edited or checked. The user must acknowledge the reminder note before proceeding.

[edit]
Suspended, Terminated and Imminent Expiry

Contracts that are Suspended, Terminated or ending within 6 weeks can only have approvals issued by user logins with 'Admin', 'Manager' or 'Accounts' role access, who must first acknowledge the validation message. If the suspend date is 'future dated' then approvals are permitted.

[edit]
Odometer Validation Flag

After editing either the 'Date' or 'Odometer' fields, a pop-up warning box will occur if the Odometer differs from the estimated value by more than the tolerance % (usually set to 20% using the gb_control fm.odometer_valid_tolerance). You will be given the choice of forcing the valid flag (checkbox) by clicking OK or un-setting the valid flag by clicking cancel. Exceptions are:

  • Odometer falls within estimated range (e.g. -20% to +20%), the flag will be silently set to valid.
  • Odometer is blank or 0, flag will be silently set to invalid.

Refer to section: Estimated Odometer Calculation for details on how odometers are estimated by the system.

 The aim of this is to force awareness/responsibility for the status of the Odometer valid flag, which in turn affects many areas of the system.

Items checked on "Save"
  • Invoice date must be a valid non-blank date (this is auto-populated with today's date but can be changed).
[edit]
Items checked to enable "Check" to appear
  • At least one approval line must be entered.
[edit]
Items checked on "Check"

In the Header area:

  1. Supplier code must be entered.
  2. Invoice number must be populated.
  3. If the Reimbursement Flag is checked ON, the Driver's "BSB Number" and "Bank Account Number" fields must be populated.
  4. Refrence number must be populated.

Each approval line:

  1. Class code must be entered.
  2. Recharge flag must be entered.
  3. Description must be entered.
  4. GST code must be non-blank
  5. GST must be '0' if GST flag(G) is set to 'N'.
  6. GST must be within the set tolerance.

In the Totals area:

  1. Validate the Total GST is within the set tolerance. Visit GST Checks for full details.
[edit]
Reimbursement Notification emailed on "Post"

If the "Reimbursement Flag" is checked ON and the Drivers / Finances "Stop BECS" field is checked OFF the user is asked: "Send Reimbursement Notification email?"
Press "OK" to see a pop-up window of the email (which can be edited) and then sent. The email is then stored in the Contract / Events screen.
If the email is cancelled in the Approvals screen, an un-sent event is stored for later use.

Reimibursement Notification emailed on "Post" will not work if gb_controls.approval_reimbursement_notification_flag is set to "no".

[edit]

Header Field - Tyres?

The Tyres? field displayed in the Contracts/Header area indicates the following: the first character of the field is the tyre limit:

  • U = Unlimited
  • N = No tyres allowed
  • # = The number show is the number of tyres that was budgeted for.


The 2nd position shows the number of tyres that have been used but not not recharged.
