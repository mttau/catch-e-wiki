Overview

This process is allows you to create or modify contract budgets in a bulk format. For an individual contract these changes should be made in the Contracts / Lease tab.
You can;

  1. Populate the budgets for a newly accepted contract.
  2. Populate the budgets for a newly created contract during an on-boarding process.
  3. Modify the budget amounts of an existing contract.
  4. Extend the number of billing periods for a contract.
  5. Change the pay cycle of an existing contract, over-write the budget amounts and extend the number of billings to match.
  6. Add budgets in bulk when Onboarding contracts.


Note: Updates will not work correctly for monthly contracts if the Client's "Common Day" and the Contract's "Common Day" do not match.

[edit]

Workflow

See also Main Page → Workflow → Fleet Management → Scheduler Jobs → Contract Budgets

Get supporting reports

  1. Catch-e Support can set-up a job to give you the reports you need
    1. Import Data / Contract Budgets - Class List
    2. Import Data / Contract Budgets - Current Budgets
  2. Class List
    1. You need a listing of all classes stored in the Contract Types / Accruals tab.
    2. Even if you don't need to update all of the budgets, or they are not used or relevant to the records being uploaded, they must all be present in the upload file, along with their companion GST columns (for all Contract Types).
  3. Current Budgets
    1. The Current Budgets report gives you a starting point to update budgets on existing leases
    2. Be aware that any 'Pending' adjustment records that exist and are dated before the next_billing_period of a contract in the upload file will be retained, and any that come after that date will be removed by this process
    3. Use the field max_item_no to identify and exclude or manually update contract records that would be flushed by a bulk update here

Create the import file

  1. Data File
  2. Using the information above, create an import file with the required record changes
  3. Import the file (via Fleet / Import / Contract Budgets) into Support to check it works correctly, then import into Live.
    1. Go to Import Data / Whole Table for the general steps for importing a file.
  4. Validation checks are done on the import file. The import will stop and you will see Validations and Alert messages if there are any issues
  5. Once the validations have passed, the file is imported and the required adjustments are made in the system automatically.


  When using this feature, we recommend testing the file by first uploading it into your Support environment to check the changes.

Importing Files

Field Type Required Comments
contract_idNumyes These number must have a matching system "Contract ID". The contracts must also contain populated "Contract Start" and "Contract End" dates.
billing_intervalTextyes Any "Periodic Interval" that is available to you on the Contracts / Billing tab of your system can be used here.
The values need to be populated in their raw database form, common values are ‘weekly’, ‘fortnightly’, 'fortnightly2' and ‘monthly’. See this query also.
next_billing_periodNumyesThis is the billing period that the budgets will be created or updated from. For new budgets, make this '1'.

For existing budgets, this can be any period that is greater than the existing posted periods. The entered number must already exist or maintain the existing sequence.
E.g. last stored period is '34', you can enter '35' but not '36' as this will leave a gap in the sequence.
Here are some examples;

  1. Periods 1-5 billed, 6-10 pending. Next billing period 5 = no - a posted record exists
  2. Periods 1-5 billed, 6-10 pending. Next billing period 12 = no - this leaves a gap in the sequence
  3. Periods 1-5 billed, 6-10 pending. Next billing period 6 = yes
  4. Periods 1-5 billed, 6-10 pending. Next billing period 10 = yes
next_billing_dateNumyesEnter the date you want to use for the next_billing_period. The subsequent periods will be dated based on the chosen billing_interval.
I.e. weekly - 7 days, fortnightly - 14 days, monthly - a month.

Currently you cannot choose 'EOM'. If the monthly date is not available in subsequent months, the last available day will be used.
E.g. If you use '31/08/2018' here, the next period will be dated '30/09/2018'.
This field must contain a valid date and it must be equal or greater to the contract's "Start Date" and be less than or equal to the "End Date".

remaining_billing_periods NumyesEnter the number of periods that should be created including the next_billing_period. E.g. '35' + '5' = '39' as the last billing period.

This field is capped at 500 to prevent problems with data processing.

Class and ClassGSTNumyesIn the file, you need a column for each "Class" that is stored in the Contract Types / Accruals tab for any Contract Type you have set-up in your system, even if it is not in use on the contracts you are updating.

The header name you use for each accrual must be the same as the stored Posting Classes / Details "Class code". E.g. FIN.
Populate each "Class" column with the Net Amount you want used for the contract.
Next to each "Class" column, you need a second column which is "Class"+ GST. E.g. FIN and FINGST. This column contains the GST Amount you want used for the contract.
Populate all of the columns with the required amounts.
If any of the classes in the file are not applicable or have a value for a particular record, populate the two columns (Class and ClassGST) as 0.
E.g. F, FGST, FIN, FINGST, INS, INSGST...


[edit]

Validation Rules

  1. All ‘contract_id’ fields in the file must match a valid system contract_id.
  2. There must be a posting class stored in the Contract Types / Accruals tab for the contract that matches each budget in the file with a value.
    (I.e. you can upload contracts with different Contract Types, but if the budget is not an accrual, it must be 0).
  3. The file must be a csv format file.

Useful Queries

[edit]
Accrual classes

Alphabetical listing of all classes stored in the Contract Types / Accruals tab. These must all be present in the upload file, along with a companion GST column. 

SELECT st1.import_column
FROM (
select pc.posting_class_code,pc.posting_class_code AS import_column
from gl_posting_maps AS pm
INNER JOIN gl_posting_class_maps AS pcm ON pcm.posting_map_id = pm.posting_map_id
AND pcm.category_type = 'accrual'
INNER JOIN gl_posting_classes AS pc ON pc.posting_class_id = pcm.posting_class_id
GROUP BY pc.posting_class_code
UNION ALL
select pc.posting_class_code,CONCAT(pc.posting_class_code,'GST') AS import_column
from gl_posting_maps AS pm
INNER JOIN gl_posting_class_maps AS pcm ON pcm.posting_map_id = pm.posting_map_id
AND pcm.category_type = 'accrual'
INNER JOIN gl_posting_classes AS pc ON pc.posting_class_id = pcm.posting_class_id
GROUP BY pc.posting_class_code

ORDER BY posting_class_code, import_column
) AS st1

Valid Pay Cycles

List the Pay Cycles that can be used in this import. 

SELECT pay_cycle_code AS billing_interval,pay_cycles_per_year 
FROM gb_pay_cycles
WHERE pay_cycle_code 
IN ('daily','weekly','fortnightly','fortnightly2','period','monthly','quarterly','bi-annual'
,'annual','bi-monthly','semi-annual','4-weekly','month-end','twice-monthly','month-day')
AND billing_flag = 'yes'
ORDER BY pay_cycles_per_year DESC
;
