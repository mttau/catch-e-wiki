Overview

The "Access" checkbox enables drivers to login to the Driver Portal.
This flag is only visible to back office users. Fleet Managers cannot see this field.
Visit Employee / Contact "Access" for the employee version of this page.

You can use the "Access" checkbox to enable users to login to;

  1. A limited set of system screens listed in the Driver page.
  2. The Driver Portal.
  3. Custom screens delivering your own data screens from your web-designers using a single signon.
[edit]

Conditions for access

  1. A valid email address must be available to activate login access.
  2. A login access can be created for any driver, however the driver's vehicle data will only be visible if Contract Types / Details "Driver Contract Access" is checked.
  3. The Drivers / Password tab is only shown to internal users when "Access" is checked.

Access Email

Whenever you check the "Access" checkbox, you will be asked if you would like to send the driver an email. If you choose Yes, an email will be created and if configured, it can have an attached User Guide.
If you have a Channel Portal, a specific email and User Guide should be configured.
The standard email message can be edited before sending if it is in Text format. The sent email is stored in the System / Mail Queue.
The email can also be generated in bulk by running Fleet / Administration / Reports → Client Login.


[edit]
Access Email Event

You can set-up a contract event email to send an access email instead of using the "Access" check box email.
This is useful if you want to provide the driver with contract information in the same email.
If this is done, contract event emails can be created, but will not send until the "Access" check box of the linked driver or employee has been checked.
If "Access" is un-checked, you will be alerted as follows; "Can not send email. Driver login access must be enabled before you can send this email."

To do this, set-up a Contract Event Email with the desired wording using the available Contract Event Email Placeholders.
Enter the Posting Classes / Details "Class code" of the access email event into [gb_controls]driver_access_check_posting_class.

Emails

The email subject and text for the driver access is stored in [gb_templates] using the name 'pdf_client_login'.
If you want to use a channel-specific email message to use with your Channel Portals, create a second record and populate [gb_templates]channel_id.

The email can be configured in plain text or html. The sender can be either hard-coded or can display details of the Client / Contact "A/C Manager".

Note: if "Email" is used as the "Driver Login", you will need to change the login placeholder used on the email, replace the placeholder #driver_id# with #driver_login# in the mail_message.

Note: Additional recipients can be nominated using [gb_template_recipients].

Email Attachments Create or update the [gb_template_attachments] record that is linked to the template above.
If you want to use a channel-specific attachment, create a second record and link it to the channel template it should be used with.

The content of the attachment can be managed through the File Import/Export menu item.

If the [gb_template_attachments]status_flag is set to 'inactive' or there is no attachment record for a template, the email will still send, but there will be no attachment.

Technical Notes about "Driver Login"

Using "Driver ID" as the login is the first method we developed and is in use by many clients. The value is stored in [gb_users]user_login as the login.

Using "Email" is the second method we developed and clients can switch to this anytime. The first email address stored in [fm_drivers]home_email is copied into [gb_users]user_login_alias and is displayed in the "Driver Login" field.

If the users "Email" has already been used as a "Driver Login" the system will revert to using the "Driver ID" as the "Driver Login". Edit and update the "Email" to use a different email address as the "Driver Login".

The field "Driver Login" displays the value stored in [gb_users]user_login_alias and if this field is blank, shows the value of [gb_users]user_login.

The login process checks both [gb_users]user_login_alias and [gb_users]user_login for a match, so either login will always work.

Useful Queries

select * from gb_templates where name = 'pdf_client_login';

select t.template_id, t.name, t.sub_name, tr.name, tr.address, tr.type, tr.name_from_field, tr.address_from_field, tr.status_flag
from gb_templates as t
inner join gb_template_recipients as tr on tr.template_id = t.template_id
where t.name = 'pdf_client_login';
