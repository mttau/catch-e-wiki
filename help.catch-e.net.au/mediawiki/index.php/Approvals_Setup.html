Access

You can access the Approvals module if the global control new_approvals_screen is set to "yes" and the Approvals menu has been configured (contact your Account Manager to configure this).
If this is turned on, the Fleet/Administration/Approvals menu path and screens are available to users and the existing Approvals screen within Fleet/Administration/Contracts becomes hidden.
Access to the screen is the same for users as the existing Contracts/Approval tab.
If you click on the "App #" hyperlink in the Contracts/Maintenance screen or the Approvals/Maintenance screen, you will be navigated to the Approvals screen that is in use for your organisation.

[edit]

Configuration - Header

approvals_contract_dates_message_flag is set to "yes" by default. The following alerts will be displayed if the business rules are triggered. These can be turned off if the flag is set to "no".

  • Contract not yet started, are you sure?
  • Contract due to end within the next 6 weeks, are you sure?
  • Contract End date has been reached, are you sure?
  • Contract has been suspended, are you sure?

approvals_franchise_optional_flag is set to "yes" by default. This enables you to use either a Supplier record or a Franchise record on your approval.
If a supplier with no franchise records attached is selected, the Address, Contact and Phone fields will be populated with information from the supplier record.
If a supplier with franchise records is selected, the Address, Contact and Phone fields will remain blank, and will be populated from a franchise record (if one is selected).
If the flag is set to "no", users will be required to select a Franchise. The Supplier lookup box will only return suppliers who have Franchise records attached to them.

[edit]

Configuration - Rows

approvals_interim_service_tolerance defines the validation tolerances allowed for an interim service. If the selected Clients/Approval Defaults "Interim Service Allowed" flag is checked off, this validation will be run and the user alerted if the validation fails.
The validation done to identify if a service is an interim one compares the last service record with the expected service date or kms, based on the values of "Service Interval(Mths)" and "Service Interval(Klms)" held in the Contracts/Vehicle screen. If one or both of these criteria is not met, an alert message is provided to the user advising them of the breach.
Open the Interim Service Validation example spreadsheet to see this working.

approvals_validate_identical_vmrs_code_against_vehicle_history_flag is set by default as "no".
If this is set to "yes", the user is alerted if the VMRS code has been used on this contract within the last 12 months. This can be useful to identify non-essential servicing.

approvals_suppress_gst_change_confirmation_flag is set by default as "no". When a user modifies the GST value of an approval row and the new value is not congruent with the associated Class GST flag set in Contract Types/Expenses-Approvals (i.e. GST = Y or N), the user is alerted that the row flag will be changed to match the entered value.
E.g. the Contract Type flag = Y and the user modifies the calculated value to show as 0.00.
If the flag is set to "yes", the user is not warned of the flag change when GST is updated, it is executed silently.

[edit]

Configuration - Save

approvals_code_fields_required is set by default as VMRS="no", Work="no', Cause="no", Type="no". If any of these settings is reset to "yes", the particular approval row code is required to be populated. This is validated on "Save". The user must fill in any blank records before the approval can be saved.

approvals_credit_limit_check_on_save_flag is set by default as "yes". This invokes the checking of the Client, Contract and Approval "Approval Limit" value and if the approval record Total (incl GST) exceeds one of these "Approval Limits", a process of emailing the client and setting the approval at 'Requested' status is invoked.
Emails are sent to recipients noted in gb_template_receipients.
If the flag is set as "no", this is all bypassed and the approval is saved at 'Active' status.

approvals_odometer_required_by_class can be set to make the Approvals / Approval "Odometer" field mandatory for each "Class" that is stored.
The user is alerted and the approval cannot be saved if the "Odometer" field is blank and any of the stored classes has been used. E.g. if the approval is for a service and the class 'S' is stored here, an odometer reading will be required.
Unless this control has already been set-up, you will need to contact your Account Manager or Catch-e Support to initially populate this control for you by running an update script like the one shown below. This one is a sample that populates the classes 'S','MAINT','M' and 'T'. 

REPLACE INTO gb_controls (module, name, effective_from, value)
VALUES ('fm', 'approvals_odometer_required_by_class', '0000-00-00', 'a:4:{i:0;s:1:"S";i:1;s:5:"MAINT";i:2;s:1:"M";i:3;s:1:"T";}');

Existing Client Data Migration

Catch-e staff must make a data update to enable the Approvals module to work effectively for you.

The following fields need to be updated in the fm_approvals table;

  1. supplier_franchise_vehicle_id (approvals must contain this to enable searching and editing)
  2. total_cost (this field is used in the Approvals/Search screen)
  3. total_gst (this field is used in the Approvals/Search screen)


[edit]

Roles

Only internal roles should be granted access to the Approvals area. Default roles that have access are: 'admin', 'user', 'demo', 'manager', 'accounts', 'state_manager', 'procurement', 'disposals', 'finance', 'operations', 'insurance', 'rm_manager', 'rm_user', 'treasury', 'senior_manager'
