Overview

This page describes how you can update Insurance Type for variant using the Vehicle to Insurance Type classification mapping rules. You will need to get the Insurance Type classifications from your insurer. This page will show you how to generate the mapping rules that can be used to bulk update Insurance Type for variants. The bulk update process will update [qt_variant_profiles]insurance_type_id where insurance_type_id is currently unassigned i.e. 0 or NULL.

Contact your Account Manager if you need help with generating the mapping rules.

[edit]

Steps

  1. Obtain Insurance Type Classifications from your Insurer
  2. Generate vehicle to insurance type mapping rules
  3. Upload it into _sch_insurance_type_mapping_rules
  4. Create a scheduler job to update [qt_variant_profiles]insurance_type_id or update existing variant profile creation job.
[edit]

Obtain Insurance Type Classifications from your Insurer

Ask your insurer for Insurance Type Classifications. Rules can be general or specific e.g. Vehicles with Power to Weight ratio over 130 are High Performance or All Polestar Electric Vehicles are Prestige

[edit]

Generate vehicle to insurance type mapping rules

Generate a set of mapping of rules as CSV file.

_sch_insurance_type_mapping_rules column definitions
ColumnData TypeDescription
sch_insurance_type_mapping_rule_uuid BIGINT Leave this blank
rule_order TINYINT Determines the order in which the rules are evaluated. Rules with a lower rule_order are processed first. The system will stop evaluating rules once a match is found for a specific vehicle. `1`, `10`, `65535` (Rules are processed in ascending order: 1, then 10, then 65535).

Allows a maximum of 65535 rules. The last rule should be a catch-all rule that applies when none of the preceding rules are applicable.

selection_group_id INT An identifier to group multiple selection criteria that should be combined with the same logical operator (`AND` or `OR`). Rules with the same `rule_order` and `selection_group_id` are evaluated together.

If a rule has a single selection criterion, this can be `1`. For combined criteria within the same `rule_order`, use same `selection_group_id` .

selection_group_operator VARCHAR(3) Specifies the logical operator to combine multiple selection criteria within the same `selection_group_id`. Possible values are `NULL`, `AND` or `OR`.
  • If a rule has only one selection criterion within a group, this field should be left `NULL`.
  • First criterion in the group doesn't need an operator. Set this to `NULL`.
  • For a rule that requires both rules (make and model for example) to match, set this to `AND` for the subsequent criteria in the same `selection_group_id`.
  • For a rule that matches either conditions set this to `OR` for the subsequent criteria in the same `selection_group_id`.
selection_field VARCHAR(100) The database field to evaluate as part of the selection rule. This should be a valid column name from these tables
  • qt_makes : Use alias make. e.g., make.name
  • qt_models : Use alias model. e.g., model.model_code
  • qt_variants : Use alias var. e.g., var.engine_type
  • qt_power_plant_types : Use alias ppt. e.g., ppt.power_plant_type_code
  • qt_fuel_type_translations : Use alias ftt. e.g., ftt. ftt.fuel_type_code
  • qt_vehicle_types : Use alias vt e.g., vt.lct_exempt_flag
selection_operator VARCHAR(10) The operator to use for the selection rule. This should be a valid SQL operator, e.g. `=`, `!=’, `LIKE`, `NOT LIKE`, `IN`, `NOT IN`, `>`, `>=`, ‘<` or ‘<=`
selection_value VARCHAR(500) The value to compare against in the selection rule. The format of this value depends on the `selection_operator`. For example:
  • `IN` and `NOT IN` operators, the value should be a comma-separated string enclosed in parentheses and single quotes, e.g. : (‘Alpina’, ‘Alpine’). The entire list should then be enclosed in double quotes, like so: "('Alpina', 'Alpine')".
  • All other operators should be an unquoted string, e.g. : electric, %PHEV%, 0, 1.8 etc
insurance_type_code VARCHAR(6) The target insurance type code to assign when the selection rule(s) are met. Typical values are `REFER`, `EV`, `HP`, `PRE`, `STD` or `NULL`.
  • All values except NULL must exist in qt_insurance_types table.
  • If it doesn’t exist in qt_insurance_types, qt_variant_profiles]insurance_type_id will be set to NULL.
  • Use NULL if you want to assign NULL to [qt_variant_profiles]insurance_type_id. For example, to exclude high value vehicles you can define a rule for high value vehicle and set this field to NULL.

[edit]

Create a scheduler job to update [qt_variant_profiles]insurance_type_id or update existing variant profile creation job.

Using the uploaded mapping rules this step will update [qt_variant_profiles]insurance_type_id where insurance_type_id is currently unassigned i.e. 0 or NULL.

[edit]
Option # 1 : Add to existing scheduler job (preferred).

Ask you Account Manager to add the following lines to the existing variant profile creation script. These lines must be at the end of the script after COMMIT; 

-- Run the Stored Proc to update qt_variant_profiles.insurance_type_id

CALL UpdateVariantInsuranceTypes();
[edit]
Option # 2 : Create a new scheduler job.

You will need to ask Catch-e Support to create a query with following content and provide you with the query_id. 

-- Run the Stored Proc to update qt_variant_profiles.insurance_type_id

CALL UpdateVariantInsuranceTypes();

Now create a scheduler job to execute this query.

  • System / Scheduler
  • New
  • Module : gb
  • Class : queries
  • Method : executeScript
  • Query ID : Query id created above
  • Run Time : This should be scheduled after the existing variant profile creation scheduler job. You should make it a child job of existing variant profile creation scheduler job.
