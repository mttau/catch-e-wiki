Overview

This page lists the standard system emails that are available to you and also the details of how to set-up particular emails.

All sent emails are stored in the Mail Queue and can be looked up there.
See also the Mail Failed page for reasons mail is not received and how to manage what you can.
Go to the Email UAT Checklist page for suggested UAT steps for new or changed emails.

System emails are usually stored in the table gb_templates.
Email recipients are usually stored in the table gb_template_recipients but are some are hard-coded.
Email attachments are usually stored in the table gb_template_attachments.

Set-up Notes

[edit]
gb_templates

The gb_template table stores the mail content, mail subject, sender name and sender email.
When the email is an event email, the sub_name must match the posting_class_code (e.g. COC = COC)
The email recipients are always drivers except for INFC and TLNC events which go to the nominated client contact.
Additional recipients can be added by populating gb_template_receipients.
If the sender name and email is not specified in gb_templates then the email will display the users email details.
The gb_templates sender information is not used for bulk emails. They use hard-coded sender information.
The gb_template.mail_message_format controls if the message needs to be in Text format or HTML format.
The gb_templates.mail_message_footer can be set to 'yes' or 'no' depending on whether a footer is to be included in the message.


[edit]
Text or HTML

You can use either Text or HTML for your email format.
Text emails are much easier to set-up and can be edited from within the Send Email pop-up windows in the system.
HTML emails can have logos and elaborate formatting, but you canâ€™t edit them from within the system, so are only good if you are sending an email that you do not want the option to edit before sending. Sometimes receiving systems do not download the images, so they can look a bit ugly.

[edit]
Text or HTML Set-up

Where a HTML template is used and a footer is required in the email there are two approaches that can be employed.

  1. The footer can be incorporated within the main HTML body. When doing so the gb_templates mail_footer_flag should be set to 'no'. The disadvantage to using this approach restricts flexibility as should the company email footer be changed each html template used in the table gb_templates would also need to be changed accordingly.
  2. Using the global control html_mail_footer places the control of footer changes to one place. Changing the Global Control will add the change to each html template as the same time.

To use option 2 there is a simple setup routine to follow:

  1. Convert the entire email text including the email footer from a word document to html.
  2. At the foot of the html markup identify the 'div' containing the footer text, and copy the complete 'div' to '/div' as the value in [gb_controls]html_mail_footer
  3. In the html markup delete the 'div' containing the footer text and replace with the placeholder #mail_footer_html# between the final '/div' and '/body'
  4. Add the html markup as mail_message for the applicable template in gb_templates and set the mail_footer_flag to 'yes'
[edit]
Placeholders

The fields mail_subject and mail_message can hold text that includes placeholders. Placeholder fields are either programmed into the email template within the application, or they are within your custom query if you're sending scheduled emails using a mailQuery job type.
Placeholders can be marked up either by using # or {. E.g. #reg_no#, or {reg_no}.

Common placeholders in older reports are as follows: reg no, driver given name, client contact # 1 name, client contact #2 name, sender user full name, email, job title, phone, mobile, direct no and fax no.

A good range of placeholders is available for Contract Email Events and Quote Event Emails.


[edit]
Channel-specific emails

Create a record in the gb_templates table. This creates the email content that is used.
If you don't set-up a channel-specific email, the email will use the generic email content.

Also visit the Driver Contract Access page if you want to set-up a channel-specific driver access email and user guide.

Standard email set-up details apply, except for the following specific settings;

  • The field channel_id should match the selected channel
