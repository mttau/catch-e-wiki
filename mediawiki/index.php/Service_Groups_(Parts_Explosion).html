Overview

Tables [fm_vmrs_groups] and [fm_vmrs_group_items] hold the relationships and values for a Service Group and its individual parts.
Another table called [fm_service_profiles] will hold the profile name. [qt_variants] can store the service profile for the variant, which can then link to its Service Group.

[edit]

Creating Approvals

When creating a new Approval using the Approval screen or the Franchise Portal, once an initial VMRS (code) is entered, the system will attempt to locate an active Service Group linked from the Contract Variant via its Service Profile. Where the relevant VMRS code exists in a Service Group the [fm_vmrs_group_items]unit_cost values will be used.

If a linked Service Profile cannot be found for the VMRS code entered, a generic Service Group may be used. These may be set up in [fm_vmrs_groups] and will not contain a Service Profile ID. The [fm_vmrs_group_items]unit_cost values for the generic Service Group will be used. Note given these are generic, typically no pricing information will be stored.

If a generic Service Group cannot be found, the VMRS code and its pricing (in [fm_vmrs_codes]) will be used to create the Approval. The [fm_vmrs_codes]unit_cost value will be used in this scenario.

These values will be multiplied by the “Qty” and the result will populate into the “Price” column in the approvals screen.

[edit]

Labour Rows

Note Labour rows will continue to populate as per existing functionality:

  • Price = “Qty” * “Retail Labour Rate” from [fm_supplier_franchise_vehicles] where populated).
  • % = “Retail Labour Discount” from [fm_supplier_franchise_vehicles] where populated).
  • Cost Price = Price * (1 - %) from [fm_supplier_franchise_vehicles] where populated).

For example, let's assume the following:

  • “Quantity” = 2.40
  • “Retail Labour Rate” = 150.00 (per hour)
  • “Retail Labour Discount” = 20.00%

Therefore:

  • Price = 360.00 (2.4 * 150.00)
  • Cost Price = 288.00 (360.00 x (100 - 20.00))
